# Phase 2 공부 문서: 안전 수동 제어와 Watchdog

## 이 문서의 목표

Phase 2에서는 원격 속도 명령을 모터에 바로 전달하지 않고, `safety_watchdog`을 반드시 통과시키는 안전 수동 제어 경로를 만들었다. 이 문서를 공부한 뒤에는 다음 질문에 자신의 말로 답할 수 있어야 한다.

1. ROS 2의 `Twist` 명령이 TurtleBot을 어떻게 움직이는가?
2. Watchdog, 속도 제한, 비상 정지는 각각 어떤 위험을 줄이는가?
3. 비상 정지를 해제한 직후 중립 명령을 요구하는 이유는 무엇인가?
4. 자동 테스트와 실제 로봇 테스트는 왜 둘 다 필요한가?

## 완성한 명령 경로

```text
키보드·웹·운영자 명령
        │  /safety/cmd_vel_in
        ▼
 safety_watchdog
  ├─ 유효 숫자 검사
  ├─ 속도 상한 적용
  ├─ 0.5초 명령 timeout
  ├─ emergency stop
  └─ 해제 후 neutral re-arm
        │  /cmd_vel
        ▼
 turtlebot3_node → OpenCR → 모터
```

핵심 원칙은 외부 조작기가 `/cmd_vel`을 직접 발행하지 않는 것이다. 모든 정상 제어 입력은 `/safety/cmd_vel_in`으로 보내고, watchdog만 `/cmd_vel`의 정상 발행자가 된다.

## 실제 검증 결과

| 검증 항목 | 결과 |
| --- | --- |
| WSL 자동 테스트 | 13개 통과, 실패 0개 |
| TB1 자동 테스트 | 13개 통과, 실패 0개 |
| 안전 출력 주기 | 약 19.997 Hz |
| 명령 timeout 설정 | 0.5초 |
| 직선 속도 상한 | 0.05 m/s |
| 회전 속도 상한 | 0.3 rad/s |
| 과속 입력 | 1.0 m/s, 2.0 rad/s |
| 제한된 출력 | 0.05 m/s, 0.3 rad/s |
| 단발 직진 실험 | 약 1.415 cm 이동 후 정지 |
| 단발 회전 실험 | 약 4.99도 회전 후 정지 |
| 텔레옵 종료 후 | 입력 발행자 0개, `/cmd_vel`은 0 |

`timeout_sec=0.5`는 설정값이다. 이번 실험에서는 정지 출력과 이동량을 확인했지만, 정지 지연이 정확히 500 ms인지 별도 계측하지는 않았다. 면접에서도 설정값과 실측값을 구분해서 말해야 한다.

## 반드시 알아야 할 개념

### 1. Twist

`geometry_msgs/msg/Twist`는 선속도와 각속도를 표현한다. TurtleBot3 Burger의 평면 이동에서 주로 사용하는 값은 다음 두 개다.

- `linear.x`: 전진·후진 속도, 단위 m/s
- `angular.z`: 좌회전·우회전 각속도, 단위 rad/s

나머지 축을 0으로 고정한 이유는 차동 구동 로봇이 옆으로 이동하거나 위아래로 움직일 수 없기 때문이다.

### 2. 안전 입력과 실제 모터 출력의 분리

`/safety/cmd_vel_in`은 아직 검증되지 않은 요청이고 `/cmd_vel`은 모터에 전달될 최종 명령이다. 이름만 나눈 것이 아니라 책임 경계를 만든 것이다.

운영 규칙은 다음과 같다.

- 조작기는 `/safety/cmd_vel_in`만 발행한다.
- watchdog은 정책을 적용한 뒤 `/cmd_vel`을 발행한다.
- `turtlebot3_node`는 `/cmd_vel`을 구독한다.
- 정상 운용에서는 watchdog 이외의 `/cmd_vel` 발행자가 없어야 한다.

다만 ROS 2 자체가 악의적이거나 잘못 설정된 노드의 `/cmd_vel` 직접 발행을 막는 것은 아니다. 이후에는 접근 제어, 네임스페이스, 배포 규칙과 모니터링으로 보강해야 한다.

### 3. Watchdog과 dead-man timeout

속도 명령은 한 번 보내면 영구히 유효한 값이 아니라 짧은 임대권으로 취급한다. 정해진 시간 안에 새 명령이 오지 않으면 watchdog이 0을 출력한다.

이 방식은 조작기 종료, 프로세스 오류, 일시적인 통신 단절처럼 명령 갱신이 끊기는 상황에서 마지막 속도가 계속 유지되는 위험을 줄인다.

### 4. 속도 제한과 잘못된 숫자 처리

watchdog은 입력을 다음 범위로 제한한다.

- `linear.x`: ±0.05 m/s
- `angular.z`: ±0.3 rad/s

상한을 넘은 유한한 값은 clamp하고, `NaN`이나 무한대는 안전한 0으로 바꾼다. 속도 제한은 조작 실수를 줄이지만 장애물 회피나 충돌 방지를 대신하지는 않는다.

### 5. Topic과 Service의 역할

- Topic: 계속 변하는 속도 명령처럼 연속 데이터에 적합하다.
- Service: 비상 정지 활성화·해제처럼 요청과 응답이 필요한 상태 변경에 적합하다.

따라서 속도는 topic으로 받고, e-stop 상태 변경은 `std_srvs/srv/SetBool` service로 구현했다. Service 성공 응답은 watchdog 내부 상태가 바뀌었다는 뜻이지, 물리 브레이크가 작동했다는 뜻은 아니다.

### 6. E-stop 상태

e-stop이 활성화되면 입력 명령과 관계없이 0을 출력한다. 현재 구현은 소프트웨어 비상 정지이므로 전원 차단형 물리 e-stop과 동일하지 않다.

### 7. 해제 후 neutral re-arm

비상 정지 중에도 조작기가 전진 명령을 계속 보내고 있을 수 있다. 단순히 e-stop 플래그만 해제하면 로봇이 즉시 다시 출발할 수 있다.

이를 막기 위해 해제 후 `WAITING_NEUTRAL` 상태로 전환한다. 0에 가까운 중립 명령을 새로 받은 뒤에만 다시 움직임을 허용한다.

중요한 구분은 다음과 같다.

- 장애 상태 해제: 이제 정지 원인이 제거되었다.
- 이동 권한 회복: 운영자가 중립을 거쳐 다시 제어하겠다는 의도를 보였다.

두 사건을 분리한 것이 이번 Phase의 핵심 설계 결정이다.

### 8. 상태 머신으로 설명하기

```text
TIMEOUT ──새 명령──> ACTIVE
ACTIVE ──명령 끊김──> TIMEOUT
ACTIVE/TIMEOUT ──e-stop──> ESTOP
ESTOP ──e-stop 해제──> WAITING_NEUTRAL
WAITING_NEUTRAL ──중립 입력──> ACTIVE
```

모든 비정상 상태의 출력은 0이다. 상태 머신으로 표현하면 조건문만 나열하는 것보다 허용되는 전이를 명확히 검토할 수 있다.

### 9. YAML 파라미터

속도 상한, timeout, 출력 주기를 코드에 고정하지 않고 `robot/safety_watchdog/config/tb1.yaml`에 둔다. 같은 코드로 로봇별 정책을 변경하고, 변경 내용을 Git diff와 코드 리뷰로 추적할 수 있다.

### 10. 테스트 피라미드

이번 Phase는 세 층으로 검증했다.

1. 순수 정책 테스트: clamp, timeout, e-stop, neutral 상태 전이를 빠르게 검증
2. ROS 노드 테스트: topic과 service 연결 및 실제 메시지 동작 검증
3. TB1 하드웨어 테스트: 바퀴, odometry, 키보드 조작과 정지 확인

자동 테스트만 통과해도 배선·OpenCR·모터 동작은 보장되지 않는다. 반대로 한 번 잘 움직였다는 관찰만으로 경계 조건의 재현성을 보장할 수 없다.

### 11. ROS_DOMAIN_ID 격리

테스트에서는 운용 중인 domain 42와 분리된 domain 142를 사용했다. 같은 네트워크의 실제 노드와 테스트 노드가 서로 발견되어 의도치 않은 모터 명령을 주고받는 일을 막기 위해서다.

### 12. 종료 검증은 endpoint와 process를 함께 본다

최종 텔레옵 검증에서는 다음 세 가지를 함께 확인했다.

- `/safety/cmd_vel_in`의 publisher count가 0
- teleop 및 임시 `ros2 topic pub` 프로세스가 없음
- `/cmd_vel` 출력이 0

노드 이름만 보거나 터미널이 닫힌 것만으로는 잔여 발행자를 확실히 배제하기 어렵다.

### 13. Odometry는 절대적인 지상 진실이 아니다

odometry는 바퀴 회전량을 적분한 추정치다. 바퀴를 공중에서 돌리거나 미끄러지면 실제 위치와 달라진다. 이번 실험에서는 누적 절대 좌표가 아니라 명령 전후의 변화량만 사용했다.

## 면접 질문과 모범 답변

### Q1. 왜 조작기에서 `/cmd_vel`로 직접 보내지 않았나요?

안전 정책을 모든 조작 방식에 일관되게 적용하기 위해서입니다. 입력 경로가 키보드에서 웹으로 바뀌어도 최종 모터 명령 전에 제한, timeout, e-stop을 한 곳에서 적용할 수 있습니다.

### Q2. Watchdog 노드 자체가 죽으면 어떻게 되나요?

watchdog 발행은 중단되지만 하위 OpenCR의 timeout 동작은 이번 Phase에서 직접 검증하지 않았습니다. 운영 환경에서는 프로세스 supervisor, heartbeat, 하위 제어기의 독립 timeout과 물리 e-stop을 함께 사용해야 합니다.

### Q3. 왜 e-stop 해제 후 중립 입력이 필요한가요?

해제 시점에도 기존 조작기가 비영점 명령을 계속 발행할 수 있기 때문입니다. 장애 해제와 이동 권한 회복을 분리해 의도하지 않은 자동 재출발을 막았습니다.

### Q4. 과속 명령을 거부하지 않고 clamp한 이유는 무엇인가요?

유한한 입력에 대해서는 방향을 유지하며 안전 상한 안에서 예측 가능한 조작을 제공하기 위해서입니다. `NaN`과 무한대처럼 의미 없는 값은 0으로 처리합니다. 정책에 따라 전체 거부를 선택할 수도 있습니다.

### Q5. timeout 계산에는 어떤 시간이 적합한가요?

시스템 시각 변경의 영향을 받지 않는 monotonic clock이 적합합니다. wall clock은 NTP 보정이나 수동 변경으로 점프할 수 있습니다.

### Q6. 자동 테스트만으로 충분하지 않은 이유는 무엇인가요?

자동 테스트는 경계 조건의 반복성과 회귀 방지에 강하지만 USB, OpenCR, 모터, 바닥 마찰은 확인하지 못합니다. 그래서 실제 로봇에서 명령부터 물리 이동과 정지까지 확인했습니다.

### Q7. odometry를 어떻게 해석했나요?

바퀴 encoder 기반 추정치이므로 실제 위치의 절대 기준으로 보지 않았습니다. 누적값 대신 같은 조건에서 명령 전후 delta와 실제 관찰을 함께 사용했습니다.

### Q8. 속도 제어에 Action보다 Topic이 적합한 이유는 무엇인가요?

속도는 높은 빈도로 갱신되는 최신값 중심의 스트림입니다. Action은 장시간 목표의 진행률과 취소가 필요할 때 적합합니다.

### Q9. e-stop에 Service를 쓴 이유는 무엇인가요?

명시적인 상태 변경 요청과 성공·실패 응답이 필요했기 때문입니다. 다만 service 응답과 실제 정지는 구분하여 출력과 하드웨어를 추가 검증했습니다.

### Q10. 여러 제어원이 동시에 명령을 보내면 어떻게 하나요?

현재는 우선순위 중재가 없다. 다음 단계에서는 `twist_mux` 같은 arbitration 계층을 두고 e-stop, 운영자, 자율주행의 우선순위와 lock 정책을 정의해야 합니다.

### Q11. `A message was lost` 경고는 무엇을 의미하나요?

관찰용 subscriber가 모든 샘플을 처리하지 못했거나 QoS·스케줄링 조건에서 일부를 놓쳤다는 뜻입니다. 안전 판단은 단일 echo가 아니라 endpoint, 연속 출력, 상태와 하드웨어 결과를 함께 봐야 합니다.

### Q12. 왜 테스트 domain을 142로 사용했나요?

실제 로봇 domain 42와 discovery를 분리해 테스트 메시지가 모터 경로에 섞이는 것을 막기 위해서입니다. 과도하게 큰 domain ID는 DDS 기본 포트 계산 범위를 벗어날 수 있어 142를 선택했습니다.

## 복습 문제

1. `linear.x`와 `angular.z`의 단위는 무엇인가?
2. watchdog timeout은 어떤 위험을 줄이는가?
3. e-stop 해제 직후 비영점 명령을 허용하면 어떤 문제가 생기는가?
4. Topic과 Service를 이번 구현에서 각각 어디에 사용했는가?
5. `/cmd_vel`의 정상 publisher와 subscriber는 누구인가?
6. odometry 절대값을 실제 이동 거리로 사용하지 않은 이유는 무엇인가?
7. 자동 테스트와 하드웨어 테스트가 각각 찾기 좋은 문제는 무엇인가?
8. 이번 Phase에서 아직 검증하지 않은 안전 항목 두 가지를 말하라.

### 정답

1. `linear.x`는 m/s, `angular.z`는 rad/s다.
2. 명령 갱신이 끊겼을 때 마지막 비영점 속도가 계속 유지되는 위험을 줄인다.
3. 조작기가 명령을 계속 보내는 중이면 해제와 동시에 의도치 않게 재출발할 수 있다.
4. 연속 속도 명령은 Topic, e-stop 상태 변경은 Service를 사용했다.
5. 정상 publisher는 `safety_watchdog`, subscriber는 `turtlebot3_node`다.
6. 공중 회전과 미끄러짐도 누적되는 encoder 기반 추정치이기 때문이다.
7. 자동 테스트는 상태 전이와 경계값 회귀, 하드웨어 테스트는 배선·OpenCR·모터·물리 움직임 문제에 강하다.
8. 예: 정확한 stop latency, 실제 Wi-Fi 단절, watchdog 강제 종료, OpenCR 독립 timeout, 물리 e-stop.

## 명령을 보지 않고 설명할 수 있는지 확인

- [ ] 안전 명령 경로를 그림 없이 설명할 수 있다.
- [ ] timeout, clamp, e-stop의 역할 차이를 설명할 수 있다.
- [ ] `WAITING_NEUTRAL`이 해결하는 위험을 예로 들 수 있다.
- [ ] 자동 테스트 13개와 실제 검증 결과를 구분해 말할 수 있다.
- [ ] odometry의 한계와 이번 delta 측정 방식을 설명할 수 있다.
- [ ] 구현한 것과 아직 구현하지 않은 것을 과장 없이 말할 수 있다.
