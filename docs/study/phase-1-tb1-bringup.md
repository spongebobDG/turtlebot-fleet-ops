# Phase 1 공부: TB1 Bringup과 LDS-02 장애 진단

## 이 문서의 목표

Phase 1에서 실행한 명령을 외우는 것이 아니라 다음 질문에 자기 말로 답할 수 있게 한다.

- ROS 2 토픽이 보이는데 데이터가 없다는 것은 무슨 뜻인가?
- 센서 문제를 어떤 순서로 좁혀야 하는가?
- UART의 TX, RX, GND는 왜 그렇게 연결하는가?
- `/dev/ttyUSB0`과 `/dev/serial0`은 무엇이 다른가?
- 하드웨어 예외를 왜 설정으로 분리해야 하는가?

## 전체 데이터 경로

```text
실제 거리
  ↓
LDS-02 센서
  ↓  3.3V UART, TX
직렬 수신 장치
  ├─ 정상 제품 경로: USB2LDS/CP2102 → /dev/ttyUSB0
  └─ TB1 우회 경로: GPIO15 UART → /dev/serial0 → /dev/ttyS0
  ↓
ld08_driver
  ↓  sensor_msgs/msg/LaserScan
/scan
  ↓
SLAM, Nav2, 장애 감지, 웹 관제
```

문제가 생기면 위에서 무작정 설정을 바꾸지 않고 데이터가 마지막으로 확인되는 지점을 찾는다.

## 반드시 알아야 할 개념 1: 노드, 토픽, Publisher와 메시지

### 핵심

- 노드는 ROS 2에서 실행되는 기능 단위다.
- 토픽은 메시지가 흐르는 이름 있는 통신 채널이다.
- Publisher는 특정 토픽에 메시지를 보낼 수 있는 끝점이다.
- Publisher가 존재해도 실제 메시지를 발행하지 않을 수 있다.

### 이렇게 설명한다

> `/scan` Publisher가 1개 있다는 것은 드라이버가 통신 끝점을 만들었다는 뜻입니다. 센서 데이터가 실제로 들어와 LaserScan 메시지를 발행한다는 뜻은 아닙니다. 그래서 `ros2 topic info` 다음에 `ros2 topic echo`나 `ros2 topic hz`로 실제 메시지를 검증해야 합니다.

### 틀린 설명

> `/scan`이 `ros2 topic list`에 보였으므로 라이다는 정상이다.

### 바로잡기

토픽 목록은 이름의 존재만 보여준다. 실제 메시지, 주기와 값까지 확인해야 정상 동작을 판단할 수 있다.

## 반드시 알아야 할 개념 2: ROS 2 QoS

### 핵심

QoS는 메시지를 얼마나 신뢰성 있게, 얼마나 오래 보관하고, 어떤 방식으로 전달할지를 정하는 정책이다.

센서 데이터는 최신 값이 중요하고 오래된 값의 재전송 가치가 낮아 `BEST_EFFORT`, `VOLATILE`을 자주 사용한다. 구독자의 QoS가 호환되지 않으면 Publisher가 있어도 메시지를 받지 못할 수 있다.

### 이렇게 설명한다

> 센서 토픽은 지연된 과거 데이터보다 최신 데이터가 중요하기 때문에 보통 SensorDataQoS를 사용합니다. `/scan`이 보이는데 echo가 안 되면 먼저 센서 QoS로 구독해 QoS 불일치를 배제합니다. 이번 장애에서는 센서 QoS로도 시간 초과돼 데이터 원본 계층을 조사했습니다.

### 실제 확인 명령

```bash
ros2 topic info /scan --verbose

timeout 10 ros2 topic echo \
  /scan sensor_msgs/msg/LaserScan \
  --qos-profile sensor_data \
  --once
```

### 이번 결과

```text
Publisher count=1
Reliability=BEST_EFFORT
SCAN_ECHO_EXIT=124
```

QoS를 맞췄는데도 메시지가 없었으므로 센서 바이트가 드라이버에 도달하지 않는 쪽으로 가설을 이동했다.

## 반드시 알아야 할 개념 3: UART TX, RX와 GND

### 핵심

- TX는 데이터를 보내는 출력이다.
- RX는 데이터를 받는 입력이다.
- 한 장치의 TX는 상대 장치의 RX에 연결한다.
- 두 장치는 전압의 기준이 되는 GND를 공유해야 한다.
- Raspberry Pi GPIO UART는 3.3V이므로 5V 신호를 직접 넣으면 안 된다.

### 이렇게 설명한다

> UART는 송신 장치의 TX를 수신 장치의 RX에 교차 연결합니다. 이번 LDS-02는 TX 전용 센서이므로 LDS-02 TX를 Raspberry Pi GPIO15 RXD에 연결하고 공통 GND를 사용했습니다. Raspberry Pi TXD는 필요하지 않았습니다.

### 틀린 설명

> 이름이 같아야 하므로 TX는 TX에 연결한다.

### 바로잡기

TX와 RX는 장치 자신의 관점에서 붙은 이름이다. 출력과 입력이 만나야 하므로 `TX → RX`가 된다.

## 반드시 알아야 할 개념 4: 장치 파일과 직렬 경로

### 핵심

Linux에서는 하드웨어 장치를 `/dev` 아래의 파일처럼 다룬다.

| 장치 | 의미 |
| --- | --- |
| `/dev/ttyACM0` | TB1에서 OpenCR USB 장치 |
| `/dev/ttyUSB0` | CP2102 USB-UART 변환기 |
| `/dev/ttyS0` | Raspberry Pi mini UART 실제 장치 |
| `/dev/serial0` | 현재 기본 UART를 가리키는 안정적인 별칭 |

### 이렇게 설명한다

> `/dev/ttyUSB0`은 CP2102라는 USB-UART 변환기를 통해 생성된 장치이고, `/dev/serial0`은 Raspberry Pi 내부 UART의 별칭입니다. TB1은 LiDAR TX가 USB2LDS 경로로 들어가지 않아 GPIO UART를 사용했고, 코드에는 실제 칩 이름인 `ttyS0`보다 역할 기반 별칭인 `/dev/serial0`을 선택했습니다.

### 주의

장치 번호는 USB 연결 순서에 따라 바뀔 수 있다. 장기적으로는 udev 규칙이나 로봇별 설정을 사용해 장치를 안정적으로 선택해야 한다.

## 반드시 알아야 할 개념 5: 시리얼 콘솔과 포트 소유권

### 핵심

운영체제가 UART를 로그인 콘솔로 사용하면 센서 드라이버와 같은 포트를 함께 사용할 수 없다.

이번 TB1에서는 다음 상태가 확인됐다.

```text
/dev/serial0 -> /dev/ttyS0
enable_uart=1
console=ttyS0,115200
```

부팅 설정 파일에는 별칭으로 다음과 같이 기록돼 있었다.

```text
console=serial0,115200
```

### 이렇게 설명한다

> 먼저 UART 하드웨어가 활성화됐는지 확인하고, 같은 포트를 커널 콘솔이나 getty가 사용하는지 확인했습니다. `/boot/firmware/cmdline.txt`에서 시리얼 콘솔만 제거하고 `enable_uart=1`은 유지했습니다. 재부팅 후 포트 그룹이 `dialout`이고 소유 프로세스가 없는지 검증했습니다.

### 왜 백업했는가

부팅 설정을 잘못 변경하면 원격 장비가 정상 부팅하지 않거나 복구 수단을 잃을 수 있다. 변경 전 파일을 백업하고 변경 범위를 한 항목으로 제한해야 한다.

## 반드시 알아야 할 개념 6: 계층별 장애 진단

### 이번에 사용한 순서

```text
ROS 토픽 존재
→ 실제 메시지와 QoS
→ 드라이버 프로세스
→ 직렬 포트 소유자
→ 프로세스 읽기 카운터
→ 원시 UART 바이트
→ 전원·GND·TX 물리 배선
```

### 이렇게 설명한다

> 상위 계층의 설정을 반복해서 바꾸지 않고 데이터 경로를 한 계층씩 내려갔습니다. Publisher는 있었지만 메시지가 없었고, 드라이버의 직렬 읽기 카운터도 증가하지 않았습니다. 원시 바이트도 없어 물리 배선을 확인했고 TX 선 이탈을 발견했습니다.

### 실무에서 중요한 이유

- 가설마다 확인할 증거가 명확하다.
- 불필요한 재설치와 설정 변경을 줄인다.
- 변경으로 새로운 문제를 만드는 위험이 줄어든다.
- 장애 보고서에서 판단 근거를 설명할 수 있다.

## 반드시 알아야 할 개념 7: 종료 코드와 검사 문맥

### 핵심

`timeout`은 제한 시간이 지나면 대상 프로세스를 종료하고 일반적으로 124를 반환한다.

### 이렇게 설명한다

> `CAPTURE_EXIT=124`만 보면 실패처럼 보이지만, 이번 명령은 지속적으로 읽는 `dd`를 5초 후 의도적으로 종료했습니다. 실제 판정은 종료 코드뿐 아니라 저장된 44,105바이트와 정상 헤더 940개를 함께 사용했습니다.

### 틀린 설명

> 종료 코드가 0이 아니므로 모든 검사가 실패했다.

### 바로잡기

명령을 어떤 목적으로 구성했는지에 따라 기대하는 종료 코드가 다를 수 있다. 검사 결과물과 함께 판단해야 한다.

## 반드시 알아야 할 개념 8: 하드코딩과 설정 분리

### 현재 상태

TB1의 동작을 빠르게 검증하기 위해 드라이버 포트를 `/dev/serial0`으로 임시 하드코딩했다.

### 왜 최종 구현이 아닌가

- TB2는 정상 USB2LDS의 `/dev/ttyUSB0`을 사용할 수 있다.
- 같은 소스를 로봇마다 직접 수정하면 버전 관리가 어려워진다.
- 장치 교체 때 다시 빌드해야 한다.
- 코드 리뷰에서 하드웨어 차이와 로직 변경이 섞인다.

### 이렇게 설명한다

> 장애 복구 단계에서는 최소 수정으로 가설을 검증했지만, 최종 구현에서는 `port`를 launch 인자나 로봇별 YAML 설정으로 분리해야 합니다. 그러면 TB1과 TB2가 같은 코드를 사용하면서 포트만 다르게 선택할 수 있습니다.

## 반드시 알아야 할 개념 9: OpenCR과 Raspberry Pi의 역할

### OpenCR

- 바퀴와 DYNAMIXEL 제어
- 배터리와 IMU 등 TurtleBot3 하드웨어 인터페이스
- Raspberry Pi와 `/dev/ttyACM0`으로 통신

### Raspberry Pi

- ROS 2 노드 실행
- LiDAR 드라이버 실행
- 네트워크와 DDS 통신
- 이후 Robot Agent와 안전 노드 실행

### 이렇게 설명한다

> OpenCR에 UART가 있다는 이유만으로 LiDAR를 연결하면 바로 `/scan`이 생성되는 것은 아닙니다. 기본 TurtleBot3 OpenCR 펌웨어에는 LDS-02 패킷을 받아 LaserScan으로 전달하는 경로가 없으므로 펌웨어 개발이 추가로 필요합니다. 이번에는 센서와 ROS 드라이버를 직접 연결할 수 있는 Raspberry Pi UART가 더 작은 변경이었습니다.

## 실제 증거로 기억할 숫자

| 증거 | 실제 값 | 의미 |
| --- | --- | --- |
| `/scan` echo | 종료 124 | 센서 QoS에서도 메시지 없음 |
| CP2102 `rchar` | 115943에서 변화 없음 | 드라이버가 새 바이트를 받지 못함 |
| CP2102 원시 검사 | 두 설정 모두 124 | USB-UART 경로에 데이터 없음 |
| GPIO 5초 캡처 | 44,105바이트 | GPIO UART에 연속 데이터 도착 |
| `54 2c` 헤더 | 940회 | LDS-02 형식의 정상 패킷 반복 |
| 배터리 | 12.06V, 86.66% | OpenCR 센서 경로 동작 |

## 30초 설명 모범 답안

> TB1 bringup에서 `/odom`과 배터리는 정상인데 `/scan` 메시지만 없었습니다. QoS, 드라이버, 직렬 읽기 순서로 확인해 CP2102에 센서 바이트가 들어오지 않는 것을 찾았습니다. 물리 점검에서 LDS-02 TX 선 이탈을 발견했고, GPIO15 RXD로 우회해 5초 동안 44,105바이트와 정상 헤더 940개를 확인한 뒤 `/scan`을 복구했습니다.

## 1분 설명 모범 답안

> TurtleBot3 Burger의 ROS 2 Humble bringup을 진행했습니다. `/scan` Publisher는 1개였지만 SensorDataQoS로 echo했을 때 시간 초과됐습니다. 그래서 ROS 설정을 반복 변경하지 않고 드라이버 프로세스와 `/dev/ttyUSB0`을 조사했습니다. 드라이버가 포트를 소유했지만 읽기 카운터가 증가하지 않았고 원시 UART도 수신되지 않았습니다. 물리 점검 결과 LDS-02 TX 선이 커넥터에서 빠져 있었습니다. 센서가 회전한 것은 전원과 PWM만 정상이라는 뜻이었습니다. LDS-02 TX를 Raspberry Pi GPIO15 RXD에 연결하고 시리얼 콘솔을 해제한 뒤 5초 동안 44,105바이트와 `54 2c` 헤더 940개를 확인했습니다. 드라이버가 `/dev/serial0`을 읽도록 최소 수정해 `/scan`을 복구했고, 다음으로 포트를 로봇별 설정으로 분리할 예정입니다.

## 3분 설명 구성

3분 답변은 다음 순서를 따른다.

1. 목적: 단일 TB1 bringup 검증
2. 증상: `/scan` Publisher는 있으나 메시지 없음
3. 가설: QoS, 드라이버, 포트, 배선
4. 증거: echo timeout, 읽기 카운터 정지, raw timeout
5. 원인: LDS-02 TX 선 이탈
6. 해결: GPIO15 UART 우회와 드라이버 포트 변경
7. 결과: 44,105바이트, 헤더 940개, `/scan` 복구
8. 한계: 임시 점퍼와 하드코딩
9. 개선: 영구 하네스와 launch 설정 분리

## 면접 질문과 모범 답변

### 1. `/scan` 토픽이 보이는데 메시지가 없다면 무엇부터 확인하겠습니까?

모범 답변:

> 먼저 `ros2 topic info --verbose`로 Publisher와 QoS를 확인하고 SensorDataQoS로 실제 메시지를 구독합니다. 메시지가 없으면 드라이버 프로세스, 포트 소유자와 로그를 확인합니다. 다음으로 프로세스의 읽기 카운터와 원시 직렬 데이터를 확인하고, 마지막으로 전원·GND·TX 배선을 점검합니다.

### 2. 왜 ROS 2를 다시 설치하지 않았습니까?

모범 답변:

> `/odom`, `/battery_state`와 다른 노드가 정상 동작했고 `/scan` Publisher도 생성됐기 때문에 ROS 2 설치 전체가 깨졌다는 증거가 없었습니다. 원시 직렬 입력이 없다는 더 직접적인 증거가 있어 하위 계층을 조사했습니다.

### 3. QoS 문제와 센서 데이터 부재를 어떻게 구분했습니까?

모범 답변:

> Publisher의 QoS를 확인하고 호환되는 SensorDataQoS로 구독했습니다. 그래도 메시지가 없었고 프로세스의 직렬 읽기 카운터도 증가하지 않아 QoS보다 앞단인 센서 입력 부재로 판단했습니다.

### 4. 라이다가 회전했는데 왜 정상으로 판단하지 않았습니까?

모범 답변:

> 회전은 전원과 PWM 경로만 검증합니다. 거리 데이터는 별도의 UART TX 경로로 전달되므로 원시 바이트와 정상 패킷 헤더까지 확인해야 합니다.

### 5. GPIO 직결의 위험과 보완책은 무엇입니까?

모범 답변:

> Raspberry Pi GPIO는 3.3V 전용이므로 신호 전압과 GND를 확인해야 하고, 느슨한 점퍼는 진동으로 빠질 수 있습니다. 검증에는 사용했지만 운영 전에는 올바른 크림프 단자나 교체 하네스로 수리해야 합니다.

### 6. 왜 OpenCR에 연결하지 않았습니까?

모범 답변:

> OpenCR에 UART 하드웨어는 있지만 기본 TurtleBot3 펌웨어가 LDS-02 패킷을 ROS LaserScan으로 전달하지 않습니다. OpenCR 펌웨어까지 수정하는 것보다 기존 Linux 드라이버가 직접 읽을 수 있는 Raspberry Pi UART를 사용하는 변경 범위가 작았습니다.

### 7. `/dev/ttyS0` 대신 `/dev/serial0`을 선택한 이유는 무엇입니까?

모범 답변:

> `/dev/serial0`은 현재 기본 UART를 가리키는 역할 기반 별칭입니다. 하드웨어 매핑이 달라져도 코드에서 실제 UART 번호를 직접 가정하는 것보다 이식성이 좋습니다.

### 8. `timeout` 124를 어떻게 해석했습니까?

모범 답변:

> 지속 실행되는 읽기 명령을 정해진 시간 후 중단하도록 만들었기 때문에 124는 예상 가능한 종료였습니다. 바이트 수와 패킷 헤더 개수를 함께 확인해 성공 여부를 판단했습니다.

### 9. TB1과 TB2의 포트가 다르면 어떻게 설계하겠습니까?

모범 답변:

> 드라이버 로직은 공유하고 `lds_port`를 launch 인자나 로봇별 YAML 설정으로 분리합니다. TB1은 `/dev/serial0`, TB2는 `/dev/ttyUSB0`을 사용하게 하고 설정 유효성 검사와 시작 로그를 추가합니다.

### 10. 이 장애에서 가장 중요한 엔지니어링 습관은 무엇이었습니까?

모범 답변:

> 추측으로 재설치하지 않고 데이터 경로를 계층별로 나눠 각 가설을 로그와 측정값으로 제거한 것입니다. 덕분에 소프트웨어 설정이 아니라 물리 TX 단선이라는 원인을 좁힐 수 있었습니다.
