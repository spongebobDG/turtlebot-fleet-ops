# 현재 프로젝트 범위와 완료 조건

결정일: 2026-07-16

## 결정

현재 MVP는 `tb1` 한 대를 대상으로 운영 가능한 수직 기능을 완성한다. 두 대를 동시에 조금씩
구현하는 대신 한 대의 bringup부터 안전 제어, 관제, 지도, 자율주행, 작업과 장애 대응까지
끝에서 끝으로 실차 검증한다.

## 현재 범위

- TB1 하드웨어 bringup과 센서·OpenCR 상태 검증
- watchdog 기반 속도 제한, timeout 정지, e-stop과 중립 재무장
- Robot Agent 상태 수집과 stale 판정
- Zenoh를 통한 WSL Fleet Gateway·웹 대시보드 연결
- SLAM 지도 작성, 저장·재로드·품질 검사
- Nav2 Goal 성공·취소·실패·timeout·재시도와 e-stop 상호작용
- 단일 로봇 작업 수명주기, 로그, 장애 감지와 운영 절차
- 자동 테스트, CI, systemd 배포, GitHub Flow, 학습·면접 문서

## 향후 확장 범위

- TB2 실차 runtime과 OpenCR·LiDAR acceptance
- 로봇별 namespace, TF tree, Action과 서비스 격리
- 두 로봇 선택·자동 작업 할당·재할당
- 공유 구역 충돌 회피와 로봇 장애 시 failover
- 다중 로봇 통합 성능·복구 테스트

기존 코드의 robot ID와 확장 가능한 인터페이스는 유지한다. 다만 위 항목은 실제 검증 전까지
완료라고 표시하지 않는다.

## TB1 MVP 완료 조건

1. 재부팅 후 필요한 서비스가 운영 절차대로 복구된다.
2. 웹에서 최신 상태와 안전 상태를 확인할 수 있다.
3. 저장 지도로 localization하고 안전 경계를 통과한 Nav2 Goal을 실행·취소할 수 있다.
4. 실패·timeout·e-stop·재시도 상태가 일관되게 기록된다.
5. 단일 로봇 작업과 장애 로그를 웹에서 추적할 수 있다.
6. 자동 테스트와 CI가 통과하고 실차 acceptance evidence가 문서화된다.

## 선택 이유

다중 로봇의 핵심은 로봇 수가 아니라 한 로봇의 상태·명령·안전·작업 경계를 명확히 만든 뒤
동일 계약을 격리해 복제하는 것이다. TB1 수직 기능을 완성하면 TB2 추가는 검증된 계약을
namespace와 robot ID로 확장하는 작업이 된다. 반대로 두 대를 동시에 진행하면 미완성 기능과
현장 변수가 늘어 근본 원인 분석과 완료 판정이 어려워진다.

## 면접 모범 답변

> 처음에는 두 대의 TurtleBot 플릿을 목표로 했지만 개발 중간에 TB1 단일 로봇 MVP로 범위를
> 조정했습니다. 기능을 줄인 것이 아니라 하드웨어, 안전 watchdog, 통신, 웹, SLAM, Nav2,
> 작업과 장애 로그를 한 대에서 운영 가능한 수준까지 끝에서 끝으로 검증하기 위한 결정입니다.
> robot ID와 확장 가능한 인터페이스는 유지하고 TB2 namespace·TF·Action 격리는 backlog로
> 분리했습니다. 따라서 현재 완료 범위와 향후 확장을 과장 없이 설명할 수 있고, TB2 추가 시
> 검증된 단일 로봇 계약을 복제할 수 있습니다.

## 복습 문제와 정답

### 1. 단일 로봇 MVP로 줄이면 플릿 프로젝트가 아닌가?

정답: 아니다. 플릿의 기본 단위인 상태, 명령, 안전, 작업 수명주기와 장애 계약을 한 로봇에서
완성하는 단계다. 다중 로봇은 검증된 계약에 격리와 할당 정책을 추가하는 확장이다.

### 2. 왜 TB2 코드를 삭제하지 않는가?

정답: 이미 확인한 확장 준비와 robot ID 계약은 재사용 가치가 있기 때문이다. 범위 밖이라는
표시와 완료 주장을 분리하면 코드를 보존하면서도 현재 성과를 과장하지 않을 수 있다.

### 3. 범위 축소의 성공 여부는 무엇으로 판단하는가?

정답: 기능 수가 아니라 명시한 TB1 완료 조건이 자동 테스트, CI, 배포 상태와 실차 증거로
모두 충족되는지로 판단한다.
