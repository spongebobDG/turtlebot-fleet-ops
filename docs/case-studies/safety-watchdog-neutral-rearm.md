# 사례 연구: Emergency Stop 해제 후 자동 재출발 방지

## 한 줄 요약

비상 정지를 해제했을 때 계속 들어오던 비영점 명령으로 로봇이 자동 재출발할 수 있는 위험을 발견하고, 중립 명령을 확인한 뒤에만 이동을 다시 허용하는 `WAITING_NEUTRAL` 상태를 추가했다.

## 상황

TB1 원격 수동 제어 경로에 속도 제한, timeout, software e-stop을 제공하는 `safety_watchdog`을 구현했다. 조작기는 `/safety/cmd_vel_in`을 발행하고 watchdog은 안전 정책 적용 후 `/cmd_vel`을 발행한다.

## 처음 구현의 문제

초기 설계는 다음과 같았다.

1. e-stop 활성화 중에는 항상 0을 출력한다.
2. e-stop 해제 시 저장된 명령을 0으로 초기화한다.
3. 이후 들어오는 새 명령은 정상적으로 허용한다.

이 설계는 단발 명령에서는 정상처럼 보인다. 그러나 키가 눌린 채이거나 웹 조작기가 동일한 전진 명령을 계속 발행하면, 해제 직후 도착한 새 비영점 메시지가 바로 통과한다. 운영자는 정지만 해제했다고 생각했는데 로봇은 자동으로 재출발할 수 있다.

## 근본 원인

“고장 상태가 해제되었다”와 “다시 움직여도 된다”를 같은 사건으로 취급한 것이 원인이었다.

- fault clear: e-stop 조건이 사라짐
- motion authorization: 운영자가 현재 입력을 중립으로 만들고 다시 제어할 의도를 보임

안전 시스템에서는 이 두 상태를 분리해야 한다.

## 해결 설계

e-stop 해제 후 즉시 정상 상태로 가지 않고 `WAITING_NEUTRAL`로 이동하도록 변경했다.

```text
ACTIVE/TIMEOUT
      │ e-stop 활성화
      ▼
    ESTOP
      │ e-stop 해제
      ▼
WAITING_NEUTRAL ──중립 입력──> ACTIVE
      │ 비영점 입력
      └──────────────────────> WAITING_NEUTRAL
```

적용 규칙은 다음과 같다.

- `ESTOP`: 입력과 관계없이 0 출력
- e-stop 해제: `WAITING_NEUTRAL` 진입, 계속 0 출력
- 비영점 입력: 재무장하지 않고 0 유지
- 중립 입력: 재무장 허용
- 재무장 뒤 새 이동 명령: 속도 제한을 적용해 전달

중립은 `neutral_epsilon` 범위 안의 `linear.x`와 `angular.z`로 판단한다.

## 왜 별도 reset 버튼 대신 중립 명령을 선택했는가

현재 목표는 키보드와 웹 조작기 모두에 공통으로 적용할 수 있는 최소 안전 게이트였다. 중립 확인 방식은 별도 UI 없이 조작 입력 자체로 “손을 놓은 상태”를 확인할 수 있다.

장점:

- 조작기 종류와 무관하게 같은 정책 적용
- e-stop 해제와 자동 재출발 분리
- 상태 전이가 단순하고 자동 테스트하기 쉬움

한계:

- 물리 e-stop이나 별도 권한 승인과 동일하지 않음
- 중립 메시지를 보낼 수 없는 고장에서는 재무장되지 않음
- 다중 운영자 환경의 승인 절차는 표현하지 못함

## 검증 방법

### 자동 테스트

정책 테스트에 다음 순서를 추가했다.

1. 비영점 명령 활성화
2. e-stop 활성화 후 출력 0 확인
3. e-stop 해제
4. 비영점 명령을 다시 넣어도 출력 0 확인
5. 중립 명령 입력
6. 그 뒤 새 비영점 명령이 허용되는지 확인

실차 배포 revision은 WSL과 TB1에서 13개 테스트가 통과했다. 최종 PR 리뷰에서
비정상 수치가 중립으로 오인되지 않는 회귀 테스트를 추가한 뒤 WSL 최종 결과는
18개 통과, 실패 0개였다.

### 최종 리뷰에서 발견한 두 번째 위험

초기 중립 판정은 입력을 먼저 정규화했다. 이때 `NaN`과 무한대가 안전한 0으로 바뀌어
중립으로 오인될 수 있었다. 재무장에서는 정규화된 출력이 아니라 원시 `linear.x`와
`angular.z`가 모두 유한한지 먼저 확인하도록 수정했다. 순수 정책 테스트와 ROS graph
통합 테스트에 비정상 수치 재무장 방지 시나리오를 추가했다.

### 실제 TB1 테스트

첫 확인에서는 e-stop 해제 시 입력 publisher가 이미 0개였다. 이 상태만으로는 “계속 발행되는 비영점 명령” 위험을 검증할 수 없다고 판단해, publisher를 유지한 채 다시 시험했다.

재시험 결과:

| 단계 | 관찰 결과 |
| --- | --- |
| e-stop 전 | 입력 publisher 1, subscriber 1 |
| e-stop 활성화 | service 성공, `/cmd_vel` 0 |
| 비영점 입력을 계속 둔 채 해제 | 입력 publisher 1 유지 |
| 중립 전 | `/cmd_vel` 0, 재출발하지 않음 |
| 입력 publisher 종료 | publisher 0 |
| 중립 명령 전송 | 재무장 완료 |
| 새 0.03 m/s 명령 | 실제 이동 후 timeout 정지 |
| 텔레옵 최종 종료 | 관련 프로세스 없음, 입력 publisher 0, 출력 0 |

## 판단 과정에서 중요했던 점

### Service 성공만으로 끝내지 않았다

`SetBool` 응답의 `success=true`는 watchdog 내부 상태 변경만 뜻한다. `/cmd_vel`이 실제로 0인지, publisher가 계속 남아 있는지, 바퀴가 정지했는지를 별도로 확인했다.

### 실패 시나리오의 전제 조건을 확인했다

첫 시험은 비영점 publisher가 사라진 뒤라 핵심 위험을 재현하지 못했다. 입력 publisher가 1개인 것을 확인한 상태로 실험을 다시 수행했다.

### 안전 범위를 과장하지 않았다

이번 개선은 watchdog을 통과하는 소프트웨어 명령 경로의 자동 재출발을 막는다. `/cmd_vel` 직접 우회 발행, watchdog 프로세스 장애, 물리 전원 차단까지 해결한 것은 아니다.

## 면접 STAR 답변

### Situation

TurtleBot 원격 제어에 software e-stop을 구현하던 중, 조작기가 비영점 속도를 연속 발행하면 e-stop 해제 직후 로봇이 다시 움직일 수 있음을 발견했다.

### Task

운영자가 정지를 해제하더라도 명시적인 안전 상태를 확인하기 전에는 이동이 재개되지 않게 해야 했다.

### Action

장애 해제와 이동 권한을 분리하고 `WAITING_NEUTRAL` 상태를 추가했다. 비영점 입력은 계속 차단하고 중립 입력을 받은 뒤에만 재무장하도록 정책을 변경했다. 정책 자동 테스트를 추가하고, 실제 TB1에서 비영점 publisher를 유지한 채 e-stop 활성화·해제·중립·재이동 순서를 검증했다.

### Result

e-stop 해제 후에도 연속 비영점 입력이 0으로 차단되었고, 유효한 중립 후 새 명령만
통과했다. 최종 WSL 테스트 18개가 통과했으며 실제 로봇도 의도한 순서로 정지하고
재이동했다.

## 1분 모범 답변

“처음에는 e-stop 해제 시 저장 명령만 0으로 초기화했습니다. 하지만 조작기가 전진 명령을 계속 발행한다면 다음 메시지가 즉시 들어와 자동 재출발할 수 있었습니다. 그래서 fault clear와 motion authorization을 분리해 `WAITING_NEUTRAL` 상태를 추가했습니다. 해제 후 비영점 명령은 계속 0으로 차단하고, 운영자가 중립 명령을 보낸 뒤에만 다시 움직임을 허용했습니다. 자동 테스트뿐 아니라 실제 TB1에서 입력 publisher가 1개로 유지된 상태를 확인하고 e-stop을 해제해도 출력이 0인지 검증했습니다. 다만 이는 software command path의 보호이며 물리 e-stop이나 우회 publisher 방지까지 해결한 것은 아닙니다.”

## 이 사례가 보여 주는 역량

- 정상 동작이 아니라 위험한 상태 전이를 먼저 찾는 능력
- 상태 머신으로 안전 정책을 명확히 표현하는 능력
- 테스트 전제 조건을 endpoint로 확인하고 부족한 실험을 다시 설계하는 능력
- 자동 테스트와 실제 하드웨어 증거를 함께 제시하는 능력
- 구현 범위와 한계를 과장하지 않는 엔지니어링 태도

## 후속 개선

1. 정확한 e-stop 및 timeout 정지 지연 계측
2. 실제 Wi-Fi 단절과 DDS participant 소실 테스트
3. watchdog 프로세스 강제 종료 장애 주입
4. OpenCR의 독립 명령 timeout 검증
5. 물리 e-stop과 전원 차단 계층 추가
6. 다중 명령원 priority·lock 정책 추가
